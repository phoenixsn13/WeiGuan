# 围观世界与身份基石 + 三愿望设计稿

日期：2026-07-02

状态：设计定稿（待用户复审 → 转 writing-plans）

本 spec 是本 batch 的唯一设计真源。它把「围观」从"单次围观观察工具"升级为"持续社交世界之上的 HITL 推演工具"，并一次性覆盖三条许愿诉求。三愿望共用同一条脊柱（`Person → Account` + 事件溯源世界），UI 是同一套调性，因此合并为一份 spec；交付按有序分片进入实现计划。

阅读前置真源：`docs/superpowers/specs/2026-07-01-weiguan-design.md`（原始产品设计）、`2026-07-01-weiguan-social-pov-ui-design.md`（社交视角 UI）、`docs/manual/2026-07-01-weiguan-cost-safety-design.md`（成本安全与注意力算法）、`HANDOFF-designer-current-state.md`（现状交接）。

## 一、背景与目标

### 现状
真围观有效性已闭合（seed 口径、random Platform、seed pinning、追问接地、成本安全 attention context）。观察工具形态基本可用。当前每次围观彼此上下文独立、发帖人由 HITL 充当、每个 Run 是一个隔离世界。

### 本轮升级的产品判断
围观本质是 **HITL 推演**：人给出初始 seed + 观察，追问抽屉已对上 OASIS 私聊。要让它产生"预演分析价值"并能作为更大仿真的 **社交媒体仿真 sidecar**，必须补齐三件事：

1. **专业社媒分析（愿望1）**：把复盘从"波次+情绪条+一段建议"升级为有方法论支撑的传播分析。这与注意力算法同构——站在社媒专业话题上做算法。
2. **持久身份与虚拟世界（愿望2）**：发帖人可选普通人/大V/KOL profile，多次发声，粉丝/立场/影响力累积。世界记住人。这是 sidecar 愿景（"买水军/买KOL"在一次宏观推演中多次执行、彼此不独立）的地基。
3. **跨平台传播（愿望3）**：一条内容在不同平台（各自 OASIS 建模）发酵走向不同；真实世界里消息在同一时间线上跨平台互窜。

### 目标
- 一套数据模型与架构同时覆盖：今天的单次围观、愿望2 的多次发声累积、愿望3 的跨平台并行。
- 保住已有的有界成本 run 引擎与注意力算法，不推翻。
- 保证前台调性全站连贯，新旧屏能拼合。

### 非目标
- 本轮不实现"内容外溢"辅通道的完整建模（截图搬运等），但数据模型须容纳它，不返工。
- 本轮不引入 LLM 型记忆摘要（确定性投影足够）。
- 本轮不重写 OASIS。

## 二、术语与心智

| 术语 | 定义 |
|------|------|
| **World（世界）** | 第一级容器。可持久、可临时。持有事件日志与投影。 |
| **Person（生物人）** | 世界里立场/认知的载体。跨平台、跨 run 持久。 |
| **Account（社媒账户）** | Person 在某平台的分身，承载该平台的粉丝/影响力。`Person → Account[]`。 |
| **EventLog（事件日志）** | 追加式，唯一真相源。任何 run/平台的产出都 append 到这里。 |
| **Projector（投影器）** | 把 EventLog 按时间线折叠成 Person/Account/关注图/分析等投影。确定性、非 LLM。 |
| **Run（推演）** | 一个平台在一段 tick 区间上的有界 OASIS 推演。今天的"一次围观"是其退化态。 |
| **WorldOrchestrator（世界编排器）** | tick 级调度：共享逻辑时钟、协调多平台并行、跨平台桥。 |
| **tick** | 世界逻辑时钟的一拍。run 占据时间线上的 tick 区间。 |

**前台心智词表**（延续既有约束，全站强制）：
- 禁止出现在普通用户主界面：`agent`、`OASIS`、`仿真`、`工作台`、`trace`、`simulation`、`dashboard`。
- 本轮新增允许词（社媒语义）：`世界`、`人物 / TA`、`身份`、`粉丝`、`影响力`、`立场`、`传播`、`发酵`、`平台`。

## 三、已锁定的三个地基决策

### 决策1 · 世界模型 = C+ 事件溯源世界编排器
在"共享时间线 + 消息互窜"的现实下：A（run 隔离）无法中途互窜、直接死；B（单一世界 DB / 单 OASIS 平台）丢掉"不同平台不同建模→不同发酵"、且 DB 无限涨；朴素 C（各 run 末尾写回可变账本）丢失发酵中途互窜且并发写覆盖。

**采用 C+**，三个升级点各解决一个隐患：
1. 身份账本改为**事件日志的投影**（event sourcing）→ 并发写变并发 append，不覆盖。
2. **tick 级世界编排器**，多平台仿真在同一逻辑时钟协调推进 → 共享时间线。
3. **跨平台桥**，每拍把可外溢事件注入他平台下一拍 → 消息互窜。

每平台仍是各自独立的 OASIS 仿真（各自 recsys/模型）→ 保住"不同平台不同发酵"，也保住有界成本 run 引擎。**今天的单次围观 = 退化态**（1 平台、1 拍序列、World 可临时）。

### 决策2 · 身份模型 = `Person → Account`
`Person`（生物人，持久身份内核：立场/记忆投影/跨平台）→ `Account`（平台分身，承载该平台粉丝/影响力）。这是唯一能无矛盾表达"一人多号"的模型（否则同一个人在两个平台像两个不相干的人，用户可感知矛盾），也正是愿望3 的地基。

### 决策3 · 持久化成本中性（量化）
沿用成本安全文档单轮上界 `I_t ≤ B + M_max + seed_panel + W_cmax·L_cmax`：
- **LLM token 增量 ≈ 0**：持久化只改 `M_max`（自我记忆）的来源——从"本 run 自己的评论"变为"跨 run 立场摘要 + 最近 K 条本人发言"。给 `person_memory_budget` 固定上界，`I_t` 上界不变，总成本仍 `≤ A·S·I_max`。
- **存储 = 线性廉价**：EventLog `O(累计动作数)`；Person/Account 投影 `O(Person数 × 平台数)`。
- **折叠 = 非 LLM 廉价**：确定性 stance 投影复用现有关键词启发式分类器。
- **可避免的 LLM 成本**：LLM 记忆摘要本轮不做。

新增配置旋钮 `person_memory_budget`，与 `attention_comment_budget` 并列。

## 四、数据模型与架构

### 4.1 与现有 canonical 的关系
现有 `weiguan/canonical/models.py` 的 `RunSnapshot`（actors/posts/replies/reactions/follows/reports/traces）保持不变，作为**单 run 快照**。本轮在其之上新增**世界层**类型，`RunSnapshot` 成为世界投影在一个 run tick 区间上的切片。前端 `model/accumulate.ts::applyDelta`（per-run 归并）被推广为世界层 fold。

### 4.2 世界层类型（设计高度，实现细节留 plan）

```
World
  world_id: str
  created_at, clock_tick: int           # 世界逻辑时钟
  persistent: bool                       # 临时(单次围观) or 持久(多次/跨平台)

Person
  person_id: str
  display_name, persona_kind            # PersonaKind: ordinary | verified | kol
                                        #   (发帖人"是什么"，独立于 tokens.POV 的"看视角"；kol persona 常配 POV.KOL 视角)
  stance_projection: StanceState        # 立场投影(确定性折叠)
  memory_projection: BoundedMemory      # 有界自我记忆(person_memory_budget)
  accounts: list[Account]

Account
  account_id: str
  person_id, platform                    # Platform 枚举(twitter/reddit/...)
  handle, avatar_seed
  num_followers, influence_score         # (Person,Platform) 投影
  standing_projection                    # 该平台历史立场/参与

WorldEvent  (EventLog 单元，追加式)
  event_id, world_id, tick, created_at
  platform, actor_account_id
  kind                                   # post|reply|reaction|follow|report|bridge_inject|seed
  payload                                # 结构化载荷(引用 canonical 动作)

FollowGraph / AnalysisProjection ...     # 均为 EventLog 的确定性投影
```

### 4.3 读/写路径（唯一正确写法）
- **写**：任何 run/平台产出 → `append(WorldEvent)`。**绝不直接可变改** Person/Account。
- **投影**：`Projector.fold(events)` 按 tick 顺序生成 Person/Account/关注图/分析投影。增量折叠 `O(Δ事件)`。
- **run 启动**：读世界投影播种（发帖人账户 standing、粉丝、起始立场）。
- **run 每拍**：产出 → append 事件；注意力生成器读"世界投影在第 T 拍切片 + 该 Person 有界自我记忆"生成 prompt（注意力算法的输入域推广）。
- **跨平台桥**：每拍把标记为可外溢的事件转成他平台的 `bridge_inject` 事件，注入下一拍种子。

### 4.4 编排器与并发
`WorldOrchestrator` 持有世界时钟，按拍推进注册的多个 Platform Sim（每个是一次有界 OASIS run）。并行 run（同拍窗口）= 跨平台推演；顺序 run（更晚拍）= sidecar 间断多次发声。并发安全由"追加事件 + 时间线顺序折叠"保证，不需要锁可变账本。

## 五、玩法与产品定义

### 5.1 愿望2 · 持久发帖人 persona 与累积

**玩法**：发起围观时，HITL 选择自己以哪个 persona 发帖：
- `ordinary`（普通人）：低起始粉丝/影响力，发酵靠内容。
- `verified`（大V）：中等起始盘，自带一定曝光。
- `kol`（KOL）：高起始盘，一发即有初始扩散，立场影响他人更强。

**产品**：同一个 Person 多次发起 → 世界记住 TA：粉丝、影响力、历史立场随每次围观累积/漂移。历史页从"按 run 列"升级为"**按人/世界聚合**"，能看到一个身份的成长轨迹。

**本 batch 边界（非 batch 内子相，而是本轮整体范围）**：schema 一步到位按 `World/Person/Account/EventLog` 建（3-ready）；填充退化到最小——发帖人享受持久累积，**围观群众 actor 本 batch 仍每次从人设池重派**（记忆投影=本 run 的退化 Person）。actor 跨 run 记忆是 batch 之外的后续增强，schema 已容纳、不返工。这既兑现"发帖人 KOL/大V 累积"，又不为愿望3 埋返工。

### 5.2 愿望1 · 专业社媒分析方法论

把 Retro 从表层展示升级为**有方法论支撑的传播分析**。所有指标在世界投影/snapshot 上**确定性计算（非 LLM）**，成本原则同注意力算法。方法论分五族，每族给出可消费产出：

| 方法族 | 学术支撑（建模思想） | 可消费产出 |
|--------|---------------------|-----------|
| **传播/级联** | 独立级联 / 线性阈值（Granovetter 阈值模型）；Hawkes 自激过程刻画转发级联 | 传播树深度/广度、级联规模、关键转发节点 |
| **意见动力学** | DeGroot、Friedkin-Johnsen；有界信心（Deffuant-Weisbuch、Hegselmann-Krause） | 立场收敛/分化预测、共识 vs 极化倾向 |
| **极化/回音室** | 立场双峰性、同质性 homophily、跨立场互动比例 | 极化指数、回音室强度、桥接不足预警 |
| **影响力/结构** | 入度、PageRank/特征向量中心性、k-core 分解 | 关键意见领袖识别、影响力排序、核心圈层 |
| **情绪/时间动力学** | stance/sentiment detection（现关键词启发式，可升级）、情绪反转拐点、发酵半衰期 | 情绪拐点、峰值到达时间、发酵曲线、半衰期 |

**产品化**：这些不是给用户看公式，而是翻译成创作者能用的洞察卡——"谁是关键传播节点""什么时候立场拐点""这条会走向共识还是极化""发酵半衰期约 N 拍"。取代当前"波次+情绪条+一段泛泛建议"的表层表达。波次改为从真实传播/情绪动力学生成，而非固定四波。

### 5.3 愿望3 · 多平台编排 + 跨平台桥 + 皮肤

**玩法**：一个世界拉起多个平台 run（各自 OASIS 建模），同一时钟并行；发帖人用不同平台账户发同一/相似内容，或内容跨平台外溢。不同平台 recsys/节奏/动作空间不同 → 发酵走向不同，这正是 OASIS 的魅力。

**产品**：新增"多平台并列时间线"视图——同一世界下各平台发酵并排，桥接事件可视化（某条内容从 X 窜到微博）。平台皮肤差异化（下 §七）。

**跨平台流转两条通道**：主通道=共享的 Person（一人多号，立场认知是一个）；辅通道=内容外溢（本轮建模为"某 Person 用另一平台账户转述 TA 看到的内容"，EventLog 容纳，Phase-1 不做完整建模）。

## 七、UI/UX 设计语言（全站统辖）

本节是给 codex 出高质量原型图的依据。要求：codex 先据本节产出逐屏原型图（imagegen），自审调性后再用高保真 skill 实现。**本节统辖全站——凡与新世界·身份模型冲突的既有屏，其改造要求在此明确带上，避免两 batch 拼不起来。**

### 7.1 产品调性一句话（POV）
> "你发出一条内容，一个**持续的社交世界**如何围观它、发酵它，并**记住你**。"

设计上的签名张力：**沉浸的社交皮肤**（熟悉、近白、平台各异）与**围观壳/世界层**（深墨+暖琥珀、克制的仪器感但绝非工作台）的对照。用户在"我是内容创作者"与"我是世界观察者"两种心智间平滑切换。

### 7.2 三寄存器（在既有双寄存器上扩展）
既有 `design/tokens.ts` 已有 `ink #14140F / cream #F7F4EC / brand #E8A13A / accent #2C4A7C` 与 sentiment（正 `#3E9B6E`/负 `#C4553B`/中 `#8A8578`/contested 琥珀渐变）。保留，新增第三寄存器：

1. **社交皮肤区**（沉浸）：近白背景、平台各异（见 §7.3）。用户"读内容/看评论"时在此。
2. **围观壳**（控制与洞察）：深墨 `#14140F`/`#0F172A`、暖琥珀 `brand`、冷靛蓝 `accent`。导航、控制条、洞察卡在此。
3. **世界层**（宏观身份/时间/跨平台）：围观壳的延伸，用于表达持久身份、立场漂移时间线、跨平台并列。视觉上比围观壳更"星图/时间线"化——暖琥珀作为身份/影响力主色，冷靛蓝作为跨平台连接线，sentiment 色仅用于立场。**世界层禁止使用图表库默认配色**，一律走本寄存器。

字体：系统无衬线优先 `-apple-system, BlinkMacSystemFont, "Segoe UI", "Noto Sans SC", sans-serif`；数字 `tabular-nums`；不用 emoji 作结构图标（用 SVG/文本，延续现状 `emoji` 字段仅作单字符字形）。

### 7.3 平台皮肤差异化规范（愿望3）
`skins/x/` 已搭 X 皮肤骨架。规范：每个平台皮肤在**社交皮肤区内**差异化，但共享同一 `posterView`/`XFeed` 数据契约（皮肤只换表现，不换数据）。

| 平台 | 视觉记号 | 密度/节奏 |
|------|---------|----------|
| 微博客（默认/微博风） | 暖色点缀、话题 `#…#`、转发链明显 | 中密度、转发驱动 |
| X | 冷蓝互动色、handle `@`、引用卡 | 高密度、快节奏 |
| Reddit/论坛 | 中性灰、楼层/投票、按 upvote 排序 | 长文、树状讨论 |

皮肤切换不改变心智词表约束。多平台并列视图中，各列用其平台皮肤，列间连接线走世界层冷靛蓝。

### 7.4 既有屏改造要求（连贯性 carry-over —— 必须随本 batch 一起改）

| 屏 | 现状 | 本轮改造要求 |
|----|------|-------------|
| **AppShell 导航** | `选圈子 / 历史记录` + "我" 头像 + "只在你发起时推演" 徽标 | 导航语义升级为世界心智：`发起 / 世界 / 历史`；"我"头像点开 = 当前身份（persona + 影响力）。徽标保留（诚实表达仅发起时推演）。 |
| **发起页（Compose/Gallery）** | 选圈子 + 写内容 + 步数 | 新增 **persona 选择**（普通人/大V/KOL，展示起始粉丝/影响力）；新增**世界选择**（新世界 or 继续已有身份的世界）；发起前展示**成本预估**（§七成本解释）。 |
| **Live** | 单平台"我看到的"、seed 固定、评论内滚、底部真实状态 | 保留主结构；顶部身份条显示发帖 persona；为多平台预留"平台切换/并列"入口（Phase-1 单平台时降级隐藏，不造假）。 |
| **TA 视角** | 仅 UI 高亮，无 `actorView` | 本轮补 `actorView(worldProjection, accountId)` 可见性裁剪：只展示 TA 合理可见内容 + TA 自身动作高亮；与追问抽屉并存（追问=私聊，视角=看世界）。 |
| **Retro** | 波次+情绪条+一段建议 | 按 §5.2 专业分析重构：真实动力学生成的传播树/立场分化/影响力排序/情绪拐点/发酵曲线；情绪筛选口径明确化（阶段主导情绪，空态诚实）。 |
| **历史页** | 按 run 平铺 + 围观热榜 | 升级为**按人/世界聚合**：一个身份的成长轨迹（累积粉丝/影响力/历次围观），run 作为该身份时间线上的节点。 |

新增屏：**身份/世界主页**（一个 Person 的跨平台身份卡 + 立场漂移时间线 + 影响力曲线 + 名下账户）、**多平台并列 Live**。

### 7.5 逐屏原型简报（供 imagegen）
每屏给 codex 一段"原型 brief"：布局骨架 + 层级 + 关键组件 + 氛围关键词。codex 据此出图、自审、再高保真实现。

- **发起页**：围观壳深墨背景。上：内容输入（社交皮肤卡预览）。中左：persona 三选一卡（普通人/大V/KOL，各显起始粉丝·影响力·一句定位）。中右：世界选择（新世界 / 继续「某身份」的世界，后者显其当前影响力）。下：圈子 + 步数 + **成本预估条**（预计参与人数/轮次/约 ¥X，自有算力 vs 付费 API 不同提示）。氛围：克制、可信、"开局设定"感，非表单堆叠。
- **Live（单平台）**：社交皮肤为主体。顶：身份条（发帖 persona + handle + 影响力）。主体：seed 固定 + 评论内滚。右：通知/热门摘要（围观壳）。底：真实状态 + `看结果`（无假 seek）。氛围：熟悉的社媒详情页。
- **Live（多平台并列）**：世界层容器。多列，每列一个平台皮肤的评论流，列头显平台。列间冷靛蓝连接线标注桥接事件（"这条从 X 窜到微博"）。顶：世界时钟/拍进度。氛围：宏观、可见"消息互窜"。
- **TA 视角**：进入后顶部 `正在从 @handle 的视角看` + `回到我看到的`。社交皮肤，但内容裁剪为 TA 可见 + TA 动作高亮。侧栏轻量上下文（TA 是谁、立场、影响力）。追问入口保留。氛围：代入、克制，不暴露全局日志。
- **Retro（专业分析）**：世界层 + 围观壳。左导航：传播树 / 立场分化 / 影响力榜 / 情绪时间线 / 数据趋势。主区按选中方法族展示真实动力学（传播树可视化、立场双峰分布、影响力条、情绪拐点曲线）。右：创作者洞察卡（关键节点/拐点/共识or极化/半衰期/风险/建议，可重生成、持久化）。氛围：专业但可读，"复盘台"非"仪表盘"，杜绝默认图表配色。
- **身份/世界主页**：世界层。上：身份卡（persona、跨平台账户、总影响力）。中：立场漂移时间线（历次围观在时间轴上，立场色带演化）。下：影响力/粉丝曲线 + 名下账户分平台明细。氛围：星图/成长档案感，暖琥珀主色。
- **历史页（按人聚合）**：围观壳。按身份分组，每组头是身份卡，组内是该身份历次围观节点（seed 摘要、平台皮肤角标、评论/转发/影响力增量）。保留围观热榜但改为世界级热度。

### 7.6 UI 验收
- 全站不出现禁用心智词。
- 三寄存器边界清晰，世界层不使用图表库默认配色。
- 平台皮肤在并列视图中可辨且共享数据契约。
- 既有六屏均按 §7.4 改造到位，与新屏调性连贯。
- 每个新增/改造屏有 desktop + mobile 原型图落盘到 manual，并有高保真实现。

## 八、契约面（设计高度，细节留 plan）

- **世界层类型**：`World/Person/Account/WorldEvent/StanceState/BoundedMemory`（后端 canonical 扩展；前端 `model/` 对应）。
- **持久化**：EventLog 存储 + Projector（后端 `weiguan/world/`）。
- **API（新增，设计高度）**：
  - `POST /api/worlds`（建世界）、`GET /api/worlds/{id}`、`GET /api/persons/{id}`（身份主页数据）。
  - 发起 run 时 `RunConfig` 增 `world_id?`、`poster_persona`、`poster_person_id?`。
  - `GET /api/runs/{id}/frames`（回放帧，兼顾交接 P0 的帧契约）——世界层 EventLog 天然就是帧源。
  - 多平台：`POST /api/worlds/{id}/orchestrate`（注册多平台并行 run）。
  - 分析：`GET /api/runs/{id}/analysis`（专业分析投影）。
- **成本**：`RunConfig` 增 `person_memory_budget`；世界级预算汇总。
- 所有新 API 遵循既有 BYOK header 约定，不落 key。

契约锁定细节（字段级）在各 plan 内以 review 锚点固化。

## 十、测试与验收原则

- **延续 seed 口径**：单平台退化态必须保持真围观有效性（`seed_interaction_count`、`seed_engaged_actor_ids` 不回归）。
- **新增世界层断言（非 LLM）**：事件追加→投影折叠确定性、并发多 run append 不覆盖、退化态与今天等价、persona 起始盘生效、累积随多次发起单调更新、分析投影确定性。
- **成本**：`person_memory_budget` 有界，单 run LLM 成本上界不变；多平台总成本=各平台之和。
- **LLM 测试**：不自跑；由用户代跑真 key，采信回传输出；不要求任何人打印/记录 key。
- **前端**：`npx vitest run`、`npx tsc -b` 通过；原型图 + 高保真实现落盘 manual。

## 十一、角色边界与非目标

- 设计/审核/规划 = 本文档作者；实现 = codex，照设计执行、不改设计、不改弱断言绕过；每 Task 一 commit 带 `Review-Anchor:`，保留既有 `# review:` 锚点、新能力打新锚点。
- 不把 UI 拉回仿真工作台；前台保持社媒心智。
- 不用前端假数据补产品逻辑；历史/Retro/身份/分析都从世界投影派生。
- 不用"全量历史喂 agent"解决质量；全量归业务层，prompt 只吃有界注意力 + 有界自我记忆。
