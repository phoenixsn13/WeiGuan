# 围观（Weiguan）— 设计文档

> 日期：2026-07-01
> 状态：已通过 brainstorming 评审，待用户 review → 进入实现计划
> 一句话：**把你的内容，先扔给一群人围观一下。** 在真正发布前，用一个"像你熟悉的 App 一样"的第一人称界面，看一群 AI 观众怎么反应，再追问他们"为什么"。

---

## 0. 背景与定位

- **底座**：OASIS（CAMEL-AI）是一个开源的社交媒体模拟器，用 LLM agent 模拟真实用户在类 Twitter / Reddit 平台上的行为，支持最多百万级 agent，内置基于兴趣与热度的推荐算法，agent 可执行 23 种动作（发帖、评论、点赞、转发、引用、关注、举报、搜索、看趋势、被访谈等）。
- **问题**：OASIS 是**基础设施**——只有 Python API 和 SQLite 事件日志，面向写脚本的研究者，没有产品界面、没有"点一下就惊艳"的一层。
- **围观 = 在 OASIS 之上做的业务应用**，目标是**大部分人能顺畅用起来、且符合产品审美**。
- **核心原子价值**：`内容进 → 一群人的模拟反应出（传播 + 情绪 + 为什么）`。

### 0.1 一个引擎，多个"外衣"（选题收敛结论）
早期讨论的四个方向——发布前内容预演 / 合成民意 / 舆情危机推演 / 社会显微镜——**不是四个产品，而是同一核心循环的四套外衣**。差异只在三个旋钮：

| 旋钮 | 含义 |
| --- | --- |
| 人群是谁 | 大众 / 你的目标客户 / 敌意的媒体+公众 |
| 关心哪个结果 | 互动热度 / 观点分布 / 是否失控 |
| 包装与语气 | 好玩的显微镜 / 严肃的仪表盘 |

因此：**做一个引擎、一个核心循环，四个方向作为后续 preset**。

### 0.2 v1 楔子
- **先好玩再实用**：入口是"好玩的预演"，结果页顺势给出可操作建议。
- **视角（POV）是产品的脊柱**（见 §2）。v1 只做**普通发帖者第一人称视角** + **一套微博客皮肤**，其余视角/皮肤"预留结构但先不做"。

## 1. 架构：稳定内核 + 可换适配器（Ports & Adapters）

```
 OASIS 引擎（每次运行 = 一种平台形态：微博客 or 论坛）
        │  原始 SQLite 事件日志
        ▼
 ┌───────────────────────────────┐
 │  Adapter → CANONICAL MODEL     │  平台中立的规范模型
 │  Actor / Post / Reply /        │
 │  Reaction / Repost / Follow /  │
 │  Report / Step                 │
 └───────────────────────────────┘
        │
        ├──► POV 透镜   （发帖者 / 平台侧 / KOL —— 对同一份数据做过滤与取景）
        │
        └──► 皮肤 Skin  （X / 微博 / Reddit / 创作者后台 —— 纯渲染层）
```

**关键约束**：
- **皮肤是纯渲染层，永不触碰引擎**。皮肤只消费 view-model，因此换皮肤是配置项而非重写。
- **皮肤的"形态"必须匹配运行的"形态"**：微博客皮肤 ↔ Twitter 模式；论坛皮肤 ↔ Reddit 模式。不能给微博客运行套一个楼中楼论坛皮肤。
- **规范模型是稳定内核**：若未来替换模拟引擎，只改 Adapter，POV 与皮肤不动。

### 1.1 分层职责
| 层 | 职责 | v1 是否实现 |
| --- | --- | --- |
| Engine 封装 | 输入 (内容, 人群, 步数)，运行 OASIS，流式吐事件 | ✅ |
| Adapter | OASIS SQLite → 规范模型 | ✅ |
| POV 透镜 | 规范模型 + 选定视角 → view-model | ✅（仅发帖者视角） |
| Skin 渲染 | view-model → 某 App 外观 | ✅（仅一套微博客皮肤） |
| Web 应用 | Python 后端 + 前端流式 | ✅ |

## 2. 视角（POV）脊柱

像一个可选角色的游戏：**实际进行时是第一人称沉浸**，复盘时切**上帝视角**。

- **进行时 = 第一人称、沉浸、且"借用熟悉界面"**：你透过一个实体的眼睛体验模拟，UI 就长成那个实体日常用的真实 App。
  - **普通发帖者** → 看起来就是你自己的 X/微博：真实感的发布框，然后看信息流与通知一条条亮起来。**（v1 唯一实现）**
  - 平台侧 → 平台运营/审核控制台（举报队列、趋势、处置）。（预留）
  - KOL → 创作者后台仪表盘（你的数据分析）。（预留）
- **复盘 = 上帝视角**：跑完后拉远，检视任何东西——传播网络、情绪随时间、任一 agent 的内心推理。
- **INTERVIEW（追问"为什么"）在两个视角里都能用**：点任意一个人即可。

> 设计理由：第一人称 = **借来的熟悉感**。不教用户新 UI，而是复刻他们每天在用的 UI，学习成本为零 = "大部分人顺畅用起来"。工程上，所有视角读**同一份**事件日志，只是不同透镜，保持应用是引擎之上薄薄一层。

## 3. 核心循环（v1）

1. **选人群** —— 圈子卡牌画廊（科技程序员群 / 饭圈 / 财经吐槽 / 育儿妈妈 / 硬核玩家…）**+**「自定义受众」（一句话描述 → 生成人物包）。每张卡 = 一份保存好的 OASIS profile 集，约 **50–150 个 agent**（小而可信、跑得快，**不追百万级**）。
2. **写内容** —— 在所选皮肤里，用真实感的发布框写。
3. **选轮次** —— 枚举预设（见 §5.3），**不会 1–2 步就结束**，让反应有发酵过程。
4. **模拟（进行时 / 第一人称）** —— 跨多个时间步，你的信息流/通知**逐条亮起**：评论刷出来、点赞数跳动、转发扩散。这是沉浸、可截图的"好玩"时刻。
5. **追问（招牌功能）** —— 点任意评论人 → **INTERVIEW**（"你为什么这么想？"）→ ta 以人设作答。
6. **复盘 / 上帝视角** —— 拉远：情绪分布、传播随时间、Top 反应、风险点，并可下钻任一 agent 的推理。这是"实用"回报，附 **1–2 条可操作建议**（"这版容易被误读成 X；把第二句改成 Y 可能更稳"）。

### 3.1 有效围观的判定（验收硬标准）

"能调用 LLM 并产出内容" **不等于** 围观成立。产品核心承诺是**人群围绕你这一条内容产生反应**，因此一次 run 是否有效，按**针对种子帖（seed）的互动**来判，而非总活动量。统一口径复用 `analysis/retro.py::compute_metrics(snapshot).totals`（其计数天然只针对 seed）：

```
seed_comments     = totals["replies"]
seed_interactions = replies + likes + dislikes + reposts + quotes + reports   （均针对 seed）
engaged_actors    = 评论/点赞/踩/转发/举报过 seed 的不同 actor
engagement_rate   = len(engaged_actors) / (人群规模 - 1)   （不含 seed 作者）
```

- **不可再降级内核**：`seed_comments >= 1` —— 必须有人对**你这一条**说话；且 seed 互动数须从初始 0 变为 >0。
- **允许独立发帖，但不得压倒围观**：只发自己的帖、从不碰 seed 的 actor 占比 **≤ 50%**。
- **降级即失败要响**：默认用轻量/免模型推荐并**保证 seed 对每个 agent 可见**（不依赖需联网下载的重推荐模型）；推荐链路构建失败必须**抛错终止**，不得静默产出"一堆各说各话的独立帖"。
- **INTERVIEW 必须接地**：只能追问**真的对 seed 有过反应**的 actor；追问基于**同一次 run** 的现场与该 actor 的真实评论/人设，不得重建人群或凭空现编；无参与者时入口置灰、后端对未参与 actor 返 404。

> 冻结的数值门槛（链路级 3-agent 与效果级 20-agent）与具体实现（推荐降级、追问接地、fixture）见 `docs/manual/2026-07-01-weiguan-remediation-2-seed-engagement.md`。

## 4. v1 范围边界（明确"先不做"）

**先不做：**
1. **只做一个视角**：只做普通发帖者第一人称。平台侧、KOL 视角只预留接口。
2. **只做一套皮肤**：只做一套微博客皮肤。微博/Reddit/创作者后台皮肤靠 canonical 层保证可插拔，v1 不画。
3. **只做小人群**：每次约 50–150 agent；百万级不在 v1。
4. **成本/延迟走轻量**：靠"小人群 + 分步流式回放"把体感做顺；API key 走 **BYOK（用户自带 key）**。

**v1 确定做：** 圈子卡牌 + 自定义受众、第一人称沉浸式发帖与实时评论刷出、点任意评论人 INTERVIEW 追问、复盘上帝视角（情绪/传播/风险 + 1–2 条建议）。

## 5. 关键参数与配置

### 5.1 人群规模
- 卡牌默认 **80 agent**；自定义可在 50–150 间选。
- 卡牌 = 预置 OASIS profile 集（可复用官方 twitter/reddit 数据集裁剪）。

### 5.2 BYOK
- 用户在设置里填自己的 API key（OpenAI / Qwen 等 OASIS 支持的 model）。key 只存前端会话/本地，后端仅代传，不落库。
- 首屏对未填 key 的用户给出清晰引导与费用预估（参考 OASIS：100 agent × 1 步 ≈ 33.5 万输入 token）。

### 5.3 轮次枚举预设（不会 1–2 步就结束）
| 预设 | 时间步 | 语义 |
| --- | --- | --- |
| 快速围观 | 6 步 | 快速看个反应 |
| 标准（推荐） | 10 步 | 让讨论正常发酵 |
| 深度发酵 | 15 步 | 完整看到转向/极化 |

> 每步 = OASIS 一个 `env.step`。首步注入用户内容（ManualAction），其后各步为 agent 的 LLMAction。activation probability 默认 0.1–0.3，兼顾成本与热闹度。

## 6. 视觉与交互设计语言

围观有**两套并置的视觉寄存器**，必须清晰区分：

### 6.1 寄存器 A —— "借来的皮肤"（进行时信息流）
**目标：看起来就是真的那个 App。** v1 复刻一套微博客外观。

- **复刻基准：按 X（Twitter）浅色信息流复刻** —— 单列时间线、圆形头像、`@handle`、正文、下方一排操作图标（评论/转发/点赞/分享）、细分割线、蓝色强调点。
- 字体：正文用系统无衬线（`-apple-system / Segoe UI / 思源黑体`）；数字（点赞/转发计数）用等宽或 tabular-nums，跳动时不抖动。
- 密度：贴近真实 App 的紧凑度，行高与内边距对齐 X 规格（头像 40px、正文 15px、次要信息 13px 灰）。
- **禁止**在皮肤区出现围观自己的品牌色/logo —— 皮肤区要"以假乱真"，品牌只存在于外层壳（寄存器 B）。

### 6.2 寄存器 B —— 围观产品壳（选人群 / 运行控制 / 复盘上帝视角）
**目标：围观自己的品牌人格，包裹在皮肤之外。**

- **概念隐喻**："围观" = 一群人围拢过来看。好奇、带一点戏谑、但克制干净。像"社会显微镜 / 小剧场"。
- **色板**：
  - 底色：近墨 `#14140F` 深色 与 米白 `#F7F4EC` 浅色 双主题（复盘偏深色显微镜感，选人群偏浅色画廊感）。
  - 品牌强调色：**暖琥珀 `#E8A13A`**（好奇、聚光灯），配一个冷对比 **靛蓝 `#2C4A7C`**（理性、数据）。
  - 情绪语义色：正向 `#3E9B6E`、负向 `#C4553B`、中立 `#8A8578`、争议/极化 `#B4552F→#E8A13A` 渐变。
- **字体**：标题用一款有性格的中文展示体（如思源宋体 Heavy / 站酷庆科黄油体做点缀），正文用思源黑体；数据用 tabular-nums。避免默认系统字堆砌，标题要有"策展"气质。
- **形状语言**：卡牌为主，8px 圆角、1px 描边、悬停轻微升起（聚光灯感 = 微投影 + 边缘暖色高光）。
- **动效（关键交互时刻）**：
  - 评论"刷出来"：逐条淡入 + 轻微上移（60–120ms 错峰），像真的有人在回。
  - 计数跳动：数字滚动 + 短促高光，不抖动布局。
  - 传播扩散：复盘里用聚光灯/涟漪从源点扩散。
  - INTERVIEW：点头像 → 侧边抽屉从右滑入，人物"被聚光"，背景压暗。

### 6.3 交互原则
- **一次只让用户做一个决定**：选人群 → 写内容 → 选轮次 → 看 → 追问 → 复盘，线性推进，进度可见。
- **进行时不可打断的沉浸**：模拟播放期间主区域是纯皮肤，控制条（暂停/加速/跳到结果）悬浮在皮肤之外的壳里。
- **追问是"戳一下就有回应"**：任何人可点，抽屉里像私聊一样问答，答案带人设口吻。
- **复盘是"拉远 + 下钻"**：默认给结论（情绪分布 + 1–2 建议），想深究再点开传播网络/单人推理。

## 8. 数据流

```
用户填 (圈子 profile 集, 内容, 轮次, BYOK)
   → 后端: oasis.make(platform=TWITTER, agent_graph=从 profile 生成)
   → env.reset()
   → step0: ManualAction(CREATE_POST, 用户内容) 注入
   → step1..N: {agent: LLMAction()}  逐步执行
   → 每步产出增量事件 → Adapter 归一化为规范模型 → 通过 SSE/WebSocket 流给前端
   → 前端: POV 透镜(发帖者) → X 皮肤渲染 → 逐条播放
   → INTERVIEW: ManualAction(INTERVIEW, 目标 agent, 问题) → 返回该 agent 回答
   → 复盘: 对完整规范模型做聚合(情绪/传播/风险) + LLM 生成 1–2 建议
```

- **错误处理**：BYOK 无效/额度耗尽 → 明确提示并暂停，不丢已生成事件；单个 agent LLM 失败 → 跳过该 agent 该步，标注但不中断整轮。
- **规范模型是快照可序列化的**，一次运行可存成一份 JSON，供复盘与"再看一遍/分享"。

---

## 9. 测试策略

- **Adapter**：golden-file 测试（已知 OASIS DB → 期望规范模型）。
- **POV 透镜 & 皮肤**：纯函数，用 fixture 规范模型做快照测试，**不需要引擎或 LLM**，快且确定。
- **轮次/聚合逻辑**：对 fixture 事件流测情绪分布、传播曲线、风险标注的计算正确性。
- **Engine 封装**：真实 LLM 端到端真跑（marker 标记），**验收按 §3.1 的 seed 互动口径**，不是"有没有产出内容"——
  - 链路级（3-agent，`@pytest.mark.llm`）：`seed_comments >= 1` 且 `seed_interactions >= 2`，且后续步含针对 seed 的互动 delta。
  - 效果级（20-agent，`@pytest.mark.llm_effect`，默认不跑）：`engagement_rate >= 0.4`、`seed replies >= 3`、独立帖 actor ≤ 50%。
  - INTERVIEW：追问评论过 seed 的 actor 答非空且不毁档；对未参与 actor 返 404。

## 10. 里程碑建议（供实现计划参考，非本 spec 承诺）

1. Engine 封装 + Adapter + 规范模型（可跑通一次并导出 JSON）。

5. 复盘上帝视角 + 建议生成。

## 附：为什么这套设计成立（一句话）
**规范模型是稳定内核，引擎/视角/皮肤都是可换适配器**——这让"多皮肤可切换""多视角""未来换引擎"都变成配置而非重写；而**第一人称借用熟悉界面**让零学习成本地"好玩且好用"同时成立。
